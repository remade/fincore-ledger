The Planner

The single write path of the ledger — how every state change is validated, ordered, and durably recorded.

Package internal/planner/  ·  audience: engineers working on or integrating with the write path  ·  altitude: architectural overview

1 · Overview & principles

The planner is the write orchestrator. Every mutation in the system — posting a transaction, authorizing a hold, reverting, amending metadata, inserting a schema — flows through exactly one of its Submit* methods. Reads never go through it; they query projections directly.

Four principles shape everything below:

PrincipleWhat it means here
Event sourcingThe append-only log_events table is the source of truth. Transactions, balances (volume deltas), holds, and metadata are derived projections written in the same transaction as the event.
BitemporalEvery event and volume delta carries both system_time (when recorded) and valid_time (when it logically occurred). See §6.
IdempotencyA two-tier cache (Redis L1 + Postgres L2) lets a client safely retry a write without double-applying it. See §5.
Single write pathAll consistency rules (balance checks, schema enforcement, sequence ordering) live in one place, so they cannot be bypassed.

The API layer authenticates the caller, runs the access policy, and unmarshals the protobuf request into typed planner inputs (PostingInput, ConvertParams, BatchIntent). The worker invokes the planner's background entry points on a ticker. Everything that changes state ends up calling the planner.

Dependencies. The planner holds a storage.Store, a batch manager (current Merkle batch), a Redis idempotency cache, an event publisher, a policy evaluator, and a logger. They are injected via Uber Fx (planner.go) and stored behind narrow interfaces so the package is unit-tested against an in-memory fake Store.

2 · The universal write lifecycle

Almost every operation follows the same shape. Understanding this one flow means understanding 90% of the planner.

  1. Validate inputs up front (so the idempotency hash never sees a nil amount).
  2. Compute the idempotency hash from the request fields (only if a key was supplied).
  3. Check the ledger exists and is not sealed.
  4. Hand off to the shared wrapper submitWithIdempotency, which:
    • does the 2-tier idempotency lookup (Redis → Postgres) and short-circuits on a hit;
    • opens a transaction inside withDeadlockRetry;
    • runs the operation's do*InTx core (the real work);
    • commits, then populates the Redis cache and publishes the event;
    • on a concurrent same-key conflict, rolls back and re-reads the winner's record.

S->>S: validatePostings() + computeIdempotencyHash() S->>DB: checkLedgerNotSealed() S->>W: work = doPostInTx alt idempotency key present W->>R: GetIdempotencyKey (L1) W->>DB: GetIdempotencyKey (L2, on L1 miss) Note over W: hit → return cached EventID (IdempotentHit) end loop withDeadlockRetry (≤ maxDeadlockRetries) W->>DB: BeginTx W->>DB: doPostInTx — validate, balance check,
appendEvent, insert tx + volume deltas + accounts alt dryRun W->>DB: Rollback (return result, no publish) else W->>DB: Commit W->>R: SetIdempotencyKey (post-commit, best-effort) W->>C: publishEvent → subscribers end end Note over W: ErrIdempotencyKeyConflict →
rollback, re-read winner → idempotent hit
Refactor. Steps 4's wrapper (submitWithIdempotency) and the shared event writer (appendEvent) were extracted in this change. Previously each of the ~10 Submit* methods repeated the lookup / retry / commit / publish / conflict block and each do*InTx repeated the batch-id + sequence + LogEventRecord + append block. See §11.

The two layers: Submit* vs do*InTx

Each operation is split in two:

  • Submit* owns the transaction lifecycle, idempotency, retry, and publish. It is the standalone entry point.
  • do*InTx is the pure in-transaction core: it validates, checks balances, and writes the event plus projections against a caller-supplied open transaction. It never begins, commits, or rolls back.

This split is what lets batch mode reuse the exact same business logic — it calls the do*InTx cores directly inside one shared transaction (see §9), so batched and standalone writes are validated identically.

4 · Double-entry & balance logic

A posting moves an amount of an asset from a source to a destination. Direction is encoded by source→destination, never by the sign of the amount (amounts must be positive). operations_shared.go · doPostInTx

Net per bucket

A single post can carry many postings. The planner first nets them per (account, asset) bucket — building netOutputs (funds leaving) and netInputs (funds arriving). This lets one operation both debit and credit the same account and have it net out.

The available-balance check

For each non-issuer source bucket, the planner verifies funds are available — inside the transaction, so there is no time-of-check/time-of-use gap:

available = posted_input − posted_output − active_holds
require:  available + net_input_to_bucket  ≥  net_output_from_bucket

active_holds is the sum of authorized-but-not-captured holds, so a posting cannot spend funds already reserved by an authorization. This same available-balance formula is shared (helper availableBalance) by authorize, convert, and revert.

Issuer accounts (configured per ledger as IssuerAccounts, e.g. a _world account) bypass the balance check — they are allowed to go negative because they are where money enters and leaves the system.

Volume deltas (the balance projection)

Balances are not stored as a single mutable number. Each posting writes two append-only volume_delta rows:

  • source: OutputDelta = amount, InputDelta = 0
  • destination: InputDelta = amount, OutputDelta = 0

A balance is then Σ InputDelta − Σ OutputDelta over the deltas (optionally as-of a point in time — see §6). A separate checkpoint builder periodically rolls deltas up for O(1) reads; the planner itself only ever appends deltas.

flowchart TD A["post: alice → bob, 100 USD"] --> B["net per (account,asset)"] B --> C{"alice an issuer?"} C -- no --> D["available(alice,USD) ≥ 100 ?"] C -- yes --> E["skip check"] D -- no --> F["ErrInsufficientFunds"] D -- yes --> G["append TransactionPosted event"] E --> G G --> H["volume delta: alice OUT 100"] G --> I["volume delta: bob IN 100"] H --> J["upsert accounts, record tx"] I --> J

Conversions are the multi-asset case: doConvertInTx writes a zero-sum 4-leg set of deltas (source asset out of source / into destination; destination asset out of destination / into source) plus an optional 2-leg slippage pair, and balance-checks the source for source_amount + slippage.

5 · Idempotency

A client may retry a request (timeout, network blip) without knowing whether the first attempt landed. An idempotencyKey makes the retry safe: the same key replays the same result instead of applying the write twice.

Two tiers

TierStoreRole
L1Redis (ik:{ledger}:{key}, 24h TTL)Fast, non-authoritative cache. A Redis failure only degrades to a slower path — it never causes a wrong answer.
L2Postgres (UNIQUE(ledger_id, idempotency_key))Authoritative. The unique constraint is also what makes concurrent same-key writes safe.

The hash guards against key reuse

The key alone is not enough: a client might reuse a key with different content. So the planner computes a SHA-256 hash over the request fields (ledger, reference, each posting's fields, sorted metadata, valid time) using a length-prefixed encoding so field boundaries are unambiguous and the hash is deterministic. On a key hit, the stored hash must match.

Lookup resultHashOutcome
L1 or L2 hitmatchesReturn cached EventID with IdempotentHit = true — no write.
L1 miss → L2 hitmatchesPopulate L1, then return the hit.
hitmismatchErrInvalidIdempotencyInput — same key, different request.
both miss—Proceed to execute; the key is recorded inside the transaction.
Redis error—Log, fall through to L2 (authoritative). Degraded, not wrong.

The concurrent-winner race

Two requests with the same key can pass the lookup simultaneously (both see a miss). Both then try to insert the key inside their transactions. The Postgres unique constraint lets exactly one commit; the other's AppendLogEvent/insert fails with ErrIdempotencyKeyConflict. The losing request rolls back and re-reads the winner's committed record, returning it as an idempotent hit — so both callers get the same successful answer and the effect is applied once.

6 · Bitemporal model

Every event and volume delta records two timelines, so the ledger can answer both "what happened" and "what did we know, and when".

DimensionSet toAnswers
system_timealways time.Now() at writeWhen did the ledger record this? (audit, ordering)
valid_timecaller-supplied, defaults to nowWhen did this logically occur? (effective date)

Because valid_time is caller-controlled it may be in the past (backdating — e.g. booking a transaction with last week's effective date) or the future (forward-dating — scheduled settlement). A balance query can then ask "as of valid_time ≤ X" to reconstruct the books as they logically stood, or "as of system_time ≤ Y" to reconstruct what was known on a given day.

Revert at effective date. A revert can be dated to the original transaction's valid_time (atEffectiveDate) rather than now, so the reversal appears in the same logical period as what it undoes. Holds, by contrast, always use valid_time = now.

7 · Concurrency & transactions

Per-ledger ordering via a row lock

Each event gets a monotonic ledger_seq. NextLedgerSeq takes SELECT … FOR UPDATE on the ledger row, computes MAX(seq)+1, and so serializes sequence allocation per ledger. This is the system's main serialization point — and its main source of contention and deadlocks under concurrent writes to the same ledger.

Refactor note. The shared appendEvent helper now acquires the sequence at append time (after validation and balance reads) rather than at the very top of each core. This shortens the window the ledger row lock is held and is deadlock-safe — the only cross-operation shared lock is the ledger row, and hold operations grab their hold-row lock first either way.

Bounded deadlock retry

A serialization deadlock surfaces as ErrDeadlock. withDeadlockRetry retries the whole transaction up to maxDeadlockRetries times with exponential backoff + jitter (10ms → capped at 5s), aborting promptly if the context is cancelled. Non-deadlock errors are returned immediately.

Hold row locks

Capture, void, and expiry re-read the hold with FOR UPDATE inside the transaction and re-check its state (voided / expired / remaining amount) before acting. This makes the hold lifecycle race-free: a void and a capture on the same hold serialize, and the loser sees the updated state and fails cleanly.

The atomic boundary

Within one transaction the planner writes the event, the transaction/hold/conversion record, the volume deltas, account upserts, metadata history, relationships, and the idempotency key — all or nothing. Only after commit does it touch external, best-effort systems:

Inside the tx (durable, atomic)After commit (best-effort, logged on failure)
log event, projections, idempotency keyRedis cache population
balance checks, schema enforcementevent publish to subscribers

A failure of a post-commit step never undoes the committed write — Postgres remains authoritative, and the Redis cache simply repopulates on the next lookup.

8 · Holds & approvals

Hold lifecycle

A hold reserves funds (counted against available balance) without moving them. It can be captured (once or partially, up to the authorized amount), voided, or auto-expired by the background sweep.

stateDiagram-v2 [*] --> Authorized: SubmitAuthorize Authorized --> Captured: SubmitCapture (≤ remaining) Captured --> Captured: partial capture again Authorized --> Voided: SubmitVoid Authorized --> Expired: ExpireHolds (past deadline) Captured --> [*] Voided --> [*] Expired --> [*] note right of Expired Worker sweep, re-reads under FOR UPDATE; no-op if already voided/expired end note

Approval workflow (multi-sig)

An intent can be parked for approval instead of executing immediately. Required approvers each sign; once all have signed, the intent executes exactly once. The flow is crash-safe via an executing intermediate state and a deterministic execution idempotency key.

stateDiagram-v2 [*] --> pending: SubmitForApproval pending --> pending: Approve (not yet quorum) pending --> executing: Approve (quorum reached) executing --> executed: intent executed + audit event executing --> expired: recovery sweep, no committed effect executing --> executed_recovered: recovery sweep finds committed effect pending --> expired: past deadline pending --> rejected: deserialize / execute failure note right of executing Guarded transitions (UpdateApprovalStateIf) prevent a recovery sweep and the live path from clobbering each other. end note
Self-approval guard: the submitter cannot approve their own intent. Recovery: an approval stuck in executing (process crashed mid-flight) is resolved by checking whether the financial effect actually committed — via the deterministic approval:{intentID} idempotency key recorded on the event — rather than blindly re-executing. This avoids both double-execution and wrongly expiring a real effect.

9 · Batch execution

A batch runs many intents together. Two modes trade atomicity against partial progress.

ModeTransactionOn failureUse when
ALL_OR_NOTHINGOne transaction for the whole batch; each intent runs via executeIntentInTx → do*InTx.The entire batch rolls back; reports FailedAt.The intents are interdependent and must commit together.
BEST_EFFORTEach intent runs as its own standalone Submit* (its own transaction + retry).That intent is marked failed; others proceed; per-intent results returned.Intents are independent and partial success is acceptable.
Why the do*InTx split matters: because ALL_OR_NOTHING calls the same cores as the standalone path, batched writes get the identical input + schema + balance validation. Batch events carry idempotency at the event level only (no per-intent key hash).

11 · Recommendations

Findings from this review. The first four were implemented in this change (behavior-preserving internal refactors verified against the planner unit-test suite — see the diff: ~440 fewer lines). The rest are proposed follow-ups.

Implemented

StatusChangeWhy
donesubmitWithIdempotency wrapperThe lookup → retry → commit → publish → conflict-resolution block was duplicated across ~10 Submit* methods. Now one wrapper; each method supplies only its hash, event type, and do*InTx closure.
doneappendEvent helperThe batch-id + NextLedgerSeq + LogEventRecord + append block was repeated in every core. Centralizing it also makes schemaVersionV1 a single source of truth and shortens the ledger-lock window.
doneavailableBalance / checkSourceBalancesThe "input − output − active holds" math lived inline in post, convert, authorize, and revert. Now one helper, consistently wrapped errors.
doneNamed constantsReplaced hardcoded literals ("sealed", deadlock retry count, schema_enforcement / active_schema_version feature keys, schema version) with package constants, per the project's "no hardcoded constants" rule.
One deliberate behavioral consolidation: idempotency lookup now runs after the ledger/sealed check for all operations. Previously post alone looked up before that check; the unified order matches every other operation and is the more conservative choice (a replay on a now-sealed ledger is rejected rather than silently returning the cached hit). The approval audit writer intentionally keeps its own event construction — its event ID must stay stable across retries, which appendEvent's fresh-ID-per-call would break.
